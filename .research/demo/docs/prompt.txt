#  SECTION 3 — Z-Set Algebra & DBSP Theory  (sub-doc: 01-zsets-and-dbsp.md)
# ══════════════════════════════════════════════════════════════

This is THE theoretical centerpiece. Cover:

## 3.1  What is a Z-Set?

  A Z-set is a function from elements to integers (weights).
    - Weight +1 = insertion, -1 = deletion
    - Z-sets form a **commutative group**: they support addition and negation
    - This group structure is what makes incrementality possible

  DBSP Paper reference: §2 "Streams and Z-sets", Definition 2.1

  Code:
    - ZWeight type (i64):           crates/dbsp/src/algebra/zset.rs:39
    - IndexedZSetReader trait:      crates/dbsp/src/algebra/zset.rs:129
    - IndexedZSet trait:            crates/dbsp/src/algebra/zset.rs:145
    - ZSetReader / ZSet traits:     same file, ~line 180-200

  Concrete types:
    - OrdIndexedZSet (in-memory, sorted)
    - FileIndexedZSet (on-disk, sorted)
    - FallbackIndexedZSet (auto-switches between memory and file)

## 3.2  The Incrementalization Theorem

  Core insight: For a query function Q and input change Δi:
    Q(i + Δi) = Q(i) + ΔQ(Δi, state)

  Because Z-sets are a group, we can compute ΔQ (the incremental version)
  without re-reading the entire input i. We only need:
    - The change Δi
    - Previously stored state (Traces)

  DBSP Paper reference: §3 "Incremental computation", Theorem 3.3

## 3.3  Traces — The State Behind Incrementality

  A Trace stores the history of changes to a collection over time.
  Aggregation operators maintain Traces so they can compute incremental updates.

  Code:
    - Trace trait:                crates/dbsp/src/trace/mod.rs
    - Spine implementation:       crates/dbsp/src/trace/spine_async.rs
    - Batches (immutable chunks): crates/dbsp/src/trace/ord/mod.rs

## 3.4  The Circuit Model

  A DBSP circuit is a directed graph of operators connected by streams.
    - Streams carry Z-sets
    - Operators transform streams (map, filter, join, aggregate, ...)
    - The Z1 operator provides 1-cycle delay (stores state between steps)
    - Nested circuits handle iteration (e.g., recursive queries)

  DBSP Paper reference: §4 "DBSP"

  Code:
    - Circuit builder:            crates/dbsp/src/circuit/circuit_builder.rs
    - Z1 delay operator:          crates/dbsp/src/operator/z1.rs (find the Z1 or DelayedFeedback)
    - Root circuit handle:        crates/dbsp/src/circuit/dbsp_handle.rs

  Diagram: Show the medallion pipeline as a circuit graph — bronze tables as inputs,
  silver views as intermediate operators, gold views as outputs, with Z1 delays
  for the stateful operators (GROUP BY, window functions).

#  SECTION 4 — Incremental Multi-Way JOINs  (sub-doc: 02-incremental-joins.md)

The medallion SQL has multi-way joins, e.g. silver_order_items_enriched joins

## 4.1  The Delta-Join Rule

  When only Δ(order_items) arrives, the join doesn't re-scan all of products/suppliers/orders.
  Instead:
    Δ(A ⋈ B) = Δ(A) ⋈ B  ∪  A ⋈ Δ(B)

  For N-way joins, this decomposes into N incremental terms.

  DBSP Paper reference: §3.2 "Joins", and the bilinear operator incrementalization

## 4.2  JoinTrace — The Implementation

  Code:
    - Join operator:              crates/dbsp/src/operator/dynamic/join.rs
    - dyn_join_incremental:       same file, ~line 481
    - dyn_join_generic:           same file, ~line 647
    - JoinTrace struct:           same file, ~line 950

  The JoinTrace operator stores one side's Trace and joins incoming deltas against it.
  For a 2-way join A ⋈ B:
    - Left side: new Δ(A) joined with stored Trace(B)
    - Right side: new Δ(B) joined with stored Trace(A)
    - Outputs are unioned (they're Z-sets, so union = addition)

#  SECTION 7 — Transactions & Commits  (sub-doc: 05-transactions.md)
# ══════════════════════════════════════════════════════════════

## 7.1  Step vs Transaction vs Checkpoint

  - **Step** = one clock tick of the root circuit. All operators fire once.
  - **Transaction** = a user-visible batch of input changes.
    - May span multiple steps if the circuit is nested.
    - Ends when step() returns true (all propagation done).
  - **Checkpoint** = persisted snapshot of the entire circuit state.
    - Optional; enables fault-tolerant restart.

  Code:
    - DBSPHandle::step():                crates/dbsp/src/circuit/dbsp_handle.rs
    - commit_transaction():              same file
    - start_commit_transaction():        same file
    - CommitProgress tracking:           same file, early lines (~line 4+)

## 7.2  The Transaction Lifecycle

  ```
  push() data to InputHandles
     ↓
  start_commit_transaction()
     ↓
  loop { step() } until returns true
     ↓
  Outputs are ready — views are consistent
     ↓
  [optional] checkpoint() → persist state
  ```

  Code:
    - Controller step loop:              crates/adapters/src/controller.rs
    - CheckpointCommitter:               crates/dbsp/src/circuit/dbsp_handle.rs:1618
    - CheckpointBuilder:                 same file, :1545

## 7.3  What "Consistent" Means

  After a transaction completes, every materialized view reflects the exact result
  you'd get by running the SQL queries on the full input in a batch system.

  This is Feldera's key guarantee: incremental computation produces identical results
  to batch recomputation. The DBSP paper proves this (§3, Theorem 3.3).

## 8.1  Traces and Spines

  Stateful operators (GROUP BY, JOIN, DISTINCT, window functions) maintain Traces.
  A Trace is implemented as a **Spine** — a tiered merge structure:

    - Level-0: newest batch (small, in-memory)
    - Level-1: merged level-0 batches (medium)
    - Level-N: progressively larger merged batches
    - Background merge tasks compact levels asynchronously

  Code:
    - Spine implementation:       crates/dbsp/src/trace/spine_async.rs
    - Batch types:                crates/dbsp/src/trace/ord/mod.rs

## 8.2  Spill-to-Disk

  When memory pressure is high, batches are written to files instead of kept in memory.

  Storage backends:
    - LocalFS:                    crates/storage/src/backend/local_fs.rs
    - S3:                         crates/storage/src/ (look for s3 or object_store)
    - Azure Blob, GCP:            same directory

  File-backed batch types:
    - FileWSet, FileIndexedZSet:  crates/dbsp/src/trace/ord/file/ (or similar)
    - FallbackWSet:               auto-switches between memory and file

## 8.3  Buffer Cache (S3-FIFO)

  A novel cache eviction algorithm from SOSP'23. Feldera uses it to manage
  in-memory pages of on-disk batches.

  Code:
    - S3FifoCache struct:         crates/buffer-cache/src/s3_fifo.rs:29
    - insert/get/remove:          same file, ~lines 111-121
    - Default 256 shards for concurrency

  How it works:
    - 3 queues: Small (admission filter), Main (frequency-based), Ghost (evicted metadata)
    - New entries go to Small; promoted to Main if accessed again
    - Ghost tracks recently evicted keys to prevent re-admission of one-hit wonders
    - Better hit rate than LRU for database workloads (see SOSP'23 paper)

## 8.4  State for gold_cancellation_impact

  Concrete walkthrough:

  1. Inner GROUP BY (category × week) → Trace stores {(category, week) → aggregate row}
  2. Window SUM(UNBOUNDED PRECEDING) → Trace stores cumulative sums per partition
  3. Window SUM(RANGE 3 WEEKS) → Trace stores sliding window state per partition
  4. Division (cancellation rate) → stateless operator, no Trace needed

  When an order is cancelled (UPDATE bronze_orders SET order_status='cancelled'):
    - The GROUP BY emits a Z-set delta: {(old_agg, -1), (new_agg, +1)} for that week
    - Window operators update cumulative and sliding sums for affected weeks
    - All downstream rows emit their own deltas
    - Total work proportional to # of affected groups, NOT total table size

## 9.1  Sharding Strategy

  Data is hash-partitioned across workers by key.
    - Each worker owns a shard of each Z-set
    - Stateless operators (map, filter) run independently per shard
    - Stateful operators (join, aggregate) require data with the same key on the same worker

  Code:
    - shard():                    crates/dbsp/src/operator/communication/shard.rs:65
    - Exchange operator:          crates/dbsp/src/operator/communication/exchange.rs:1536
    - ExchangeSender:             same file, :1173
    - ExchangeReceiver:           same file, :1324

## 9.2  Exchange (Shuffle) Operations

  When a GROUP BY key doesn't match the current partition key, an Exchange reshuffles:
    - Each worker sends records to the worker that owns the target key
    - Uses hash(key) % num_workers for routing
    - Implemented with crossbeam channels (lock-free, bounded)

## 9.3  Commit Coordination

  All workers must complete each step before the next step begins.
  Barrier synchronization between steps ensures consistency.

## 10.1  How Buffering Works

  Instead of flushing every Z-set delta to the output transport immediately,
  the output buffer accumulates changes and flushes:
    - When the buffer time limit expires (10s in the demo)
    - When a commit completes
    - When the buffer is full

  Code:
    - Output buffer config:       crates/adapters/src/controller.rs (search for enable_output_buffer)

## 10.2  The pipeline_complete Signal

  demo.sh polls `GET /v0/pipelines/medallion/stats` and checks
  `global_metrics.pipeline_complete`.

  This becomes true when:
    - All input endpoints have reported they've read everything (no more data)
    - All buffered data has been processed through the circuit
    - All output buffers have been flushed

  Code:
    - pipeline_complete():        crates/adapters/src/controller.rs:1290

  For snapshot-mode Delta tables: "read everything" means the full snapshot has been read.
  For follow-mode: pipeline_complete stays false (always listening for new versions).

# ══════════════════════════════════════════════════════════════
#  SECTION 11 — Fault Tolerance & Checkpointing
#  (sub-doc: 09-fault-tolerance.md)
# ══════════════════════════════════════════════════════════════

## 11.1  What Gets Checkpointed

  A checkpoint captures:
    - All Trace state (the accumulated history in every stateful operator)
    - Input resume metadata (e.g., Delta table version number, Kafka offsets)
    - Output statistics (for deduplication on replay)
    - Step number (transaction boundary marker)

  Code:
    - Checkpointer struct:        crates/dbsp/src/circuit/checkpointer.rs:34
    - CheckpointBuilder:          crates/dbsp/src/circuit/dbsp_handle.rs:1545
    - CheckpointCommitter:        same file, :1618

## 11.2  Recovery Flow

  On restart after a crash:
    1. Load latest checkpoint
    2. Restore Trace state into all stateful operators
    3. Resume input endpoints from stored metadata (e.g., Delta version N)
    4. Output endpoints skip already-committed data using step numbers
    5. Processing continues from exactly where it left off — no duplicates, no gaps

## 11.3  Limitations

  Fault tolerance is a preview feature. It requires:
    - Input connectors that support replay (Delta Lake, Kafka — yes; HTTP push — no)
    - Output connectors that support idempotent writes or deduplication
    - Checkpoint storage (local disk or cloud backend)
